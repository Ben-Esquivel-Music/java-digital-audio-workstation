# Migrate the Built-In Plugin Views Onto the Editor Contract and Delete the PluginViewController Switch

## Motivation

After story 301 (Phase 2), third-party plugins render through `PluginEditorFactory`, but the built-ins are still the *coincidence of layouts glued together by a `switch` over hard-coded ids* the Design Book §1.9 calls "the gap". The remaining §1 problems are all on the built-in side:

- **§1.1 — three things called "plugin view".** Verified: there are **thirteen** hand-rolled `*PluginView.java` classes in `daw-app/.../ui/` (`BusCompressorPluginView`, `MultibandCompressorPluginView`, `DeEsserPluginView`, `TruePeakLimiterPluginView`, `TransientShaperPluginView`, `NoiseGatePluginView`, `ConvolutionReverbPluginView`, `ExciterPluginView`, `BinauralMonitorPluginView`, `MidSideWrapperPluginView`, `DitherPluginView`, `MatchEqPluginView`, `ArpeggiatorPluginView`), each a bespoke `BorderPane` with no shared scaffolding.
- **§1.5 — built-ins set a misleading example.** A developer who reads `BusCompressorPluginView` to learn "how should I shape my plugin GUI?" learns a pattern the SDK does not accept from outside the tree (§1.5).
- **§9 rejection #5 / #8 — `switch` on plugin id is the bug, not the feature.** `PluginViewController.openBuiltInPluginView()` (`daw-app/.../ui/PluginViewController.java:161-183`) dispatches fourteen arms by `PLUGIN_ID` plus a three-id arm that routes to `host.switchToMasteringView()`. This is one step shy of reflective `Class.forName("…View")` discovery (§9 rejection #8) and must go.

This is **Phase 3 of the §8 migration path** — *built-ins migrate to the contract*, one at a time, in order of triviality (§8.3). **After each step the legacy `switch` arm and its hand-rolled view class are deleted; the `switch` shrinks monotonically; when the last arm is removed, the dispatcher is deleted.** The end state honours §9 rejection #5: **no `switch` on plugin id anywhere in `daw-app`** — the only place an id drives behaviour is the registry.

## Goals

- **Migrate built-ins to `editorFactory()` in the §8.3 order**, deleting each `switch` arm and its view class as it lands:
  1. `DitherPlugin` (smallest) → `Declarative`.
  2. `NoiseGatePlugin`, `DeEsserPlugin`, `BusCompressorPlugin` → `Declarative` (their views are essentially parameter grids today).
  3. `TruePeakLimiterPlugin`, `TransientShaperPlugin`, `MultibandCompressorPlugin` → `Panel` (custom layouts, no rendered graphics).
  4. `ConvolutionReverbPlugin`, `ExciterPlugin`, `MidSideWrapperPlugin` → `Panel`.
  5. `BinauralMonitorPlugin`, `MatchEqPlugin`, `SoundWaveTelemetryPlugin`, `SpectrumAnalyzerPlugin` → `Canvas` with `ChromePolicy.IMMERSIVE`.
  6. `VirtualKeyboardPlugin`, `TunerPlugin` → `Panel` with `EditorHints.chrome() == MINIMAL` (the §5.C compact strip is a *future* enhancement, not required here).
  7. `ArpeggiatorPlugin` → `Panel`. It already ships `ArpeggiatorPluginView.java` but, unlike the others, has **no arm in the `switch` today** (verified — it is absent from `PluginViewController.java:163-181`); fold it in here so it gains a contract-driven editor and the orphaned view class is retired.
- **Delete each `switch` arm and the matching `open…Window(...)` helper as its plugin migrates.** Note the backing view class is **not uniformly a `*PluginView.java`** (§8.3):
  - Items 1–4, plus `BinauralMonitorPlugin`/`MatchEqPlugin`/`ArpeggiatorPlugin`, are backed by the thirteen `*PluginView.java` classes — delete the class with its arm.
  - `SpectrumAnalyzerPlugin`, `SoundWaveTelemetryPlugin`, `VirtualKeyboardPlugin`, and `TunerPlugin` are backed by **bespoke view classes** opened through `openSpectrumAnalyzerWindow(...)` / `openSoundWaveTelemetryWindow(...)` / `openVirtualKeyboardWindow(...)` (which builds `KeyboardProcessorView`) / `openTunerWindow(...)` (verified at `PluginViewController.java:185-263`). Those `open…Window` helpers — and the `SpectrumDisplayWindow` / `TunerDisplayWindow` standalone windows where no longer needed — are deleted alongside the arm.
- **Drop the three mastering-route arms.** `ParametricEqPlugin.PLUGIN_ID`, `CompressorPlugin.PLUGIN_ID`, and `ReverbPlugin.PLUGIN_ID` do not open an editor at all — they route to `host.switchToMasteringView()` (verified at `PluginViewController.java:178-180`). These arms are **simply dropped** from the `switch`; these three plugins surface through the mastering view as before (no contract editor is created for them by this story).
- **Delete `PluginViewController.openBuiltInPluginView()` and, when its last caller falls away, the dispatcher's reason to exist.** When no arm remains, the method is removed; the `switch` is gone from `daw-app`. The activation entry point (`onActivateBuiltInPlugin`, `PluginViewController.java:87-114`) routes built-ins through the same `editorFactory()` path story 301 built for third parties — built-ins are no longer second-class (§2.3).
- **Built-ins become reference examples** (§1.5 reversed): each migrated built-in is now a worked example of the contract a third party can copy — a `Declarative` list, a `Panel` returning a `Region`, or a `Canvas` with an `IMMERSIVE` render loop.
- Tests:
  - `NoSwitchOnPluginIdTest` (new): a source-scan / reflective assertion that `PluginViewController` no longer contains a `switch` over `PLUGIN_ID` (and that `openBuiltInPluginView` is gone) — pins §9 rejection #5. Model it on the existing UI-overhaul conformance-sentinel source-scan pattern (`feedback_ui_overhaul_conformance_sentinel.md`): comment-strip, non-empty guard, self-exclude.
  - `BuiltInDeclarativeEditorsTest` (new): `DitherPlugin`, `NoiseGatePlugin`, `DeEsserPlugin`, `BusCompressorPlugin` each return a `Declarative` factory whose parameters equal `getParameters()`; activating each yields the host parameter grid.
  - `BuiltInPanelEditorsTest` (new): `TruePeakLimiterPlugin`, `TransientShaperPlugin`, `MultibandCompressorPlugin`, `ConvolutionReverbPlugin`, `ExciterPlugin`, `MidSideWrapperPlugin`, `VirtualKeyboardPlugin`, `TunerPlugin`, `ArpeggiatorPlugin` each return a `Panel` whose `createPanel(...)` yields a non-null `Region`; the two MINIMAL-chrome ones report `ChromePolicy.MINIMAL`.
  - `BuiltInImmersiveCanvasEditorsTest` (new): `BinauralMonitorPlugin`, `MatchEqPlugin`, `SoundWaveTelemetryPlugin`, `SpectrumAnalyzerPlugin` each return a `Canvas` factory reporting `ChromePolicy.IMMERSIVE`; `attach`→`render` is exercised.
  - `MasteringRouteArmsDroppedTest` (new): activating `ParametricEqPlugin`/`CompressorPlugin`/`ReverbPlugin` no longer dispatches through a `switch` arm and does not build a contract editor — they reach the mastering view through the surviving non-`switch` path.
  - `ArpeggiatorOrphanViewRetiredTest` (new): `ArpeggiatorPluginView` is deleted and `ArpeggiatorPlugin` now yields a `Panel` editor — proves the orphaned (never-in-`switch`) view is folded in, not stranded.
  - `RemovedViewClassesGoneTest` (new): the deleted `*PluginView.java` classes and `SpectrumDisplayWindow`/`TunerDisplayWindow` (where retired) are absent — a guard against a half-migration that leaves dead bespoke views (per `feedback_test_only_callers_are_not_live_usage.md`, API kept alive only by its own tests is dead; migrate still-valid tests onto the contract).

## Non-Goals

- **No SDK changes.** The contract is frozen as of story 300; this story only re-shapes built-ins to fit it.
- **No new chrome.** The breadcrumb, footer, A/B, fault banner are all from story 301's `EditorFrame`; built-ins inherit them by switching to the contract — they must not re-implement per-view A/B (the duplication §6.5 calls out) now that `ABComparison` is surfaced uniformly in host chrome.
- **No §5.C compact strip.** `VirtualKeyboardPlugin`/`TunerPlugin` migrate as MINIMAL-chrome `Panel`s; the embed-in-the-mixer-channel compact strip (§5.C) is explicitly a future enhancement (§8.3 item 6), not part of this story.
- **No behaviour change for the mastering-route plugins.** `ParametricEqPlugin`/`CompressorPlugin`/`ReverbPlugin` keep routing to the mastering view; this story removes their `switch` arms, it does not give them contract editors.
- **No install-UX change.** `PluginManagerDialog` is untouched (story 303).
- **No `PluginUI` removal.** Story 304. (`PluginUI` is still uncalled; the built-ins migrate to `editorFactory()`, never to the deprecated interface.)
- **No bundling of tangential cleanup.** If migrating a built-in surfaces an adjacent issue (e.g. a stale processor seam), flag it against the backlog and ship the narrowly-scoped migration (`feedback_scope_fix_defer_tangential.md`).

## Technical Notes

- **Migration order is risk-ordered and each step is independently shippable** (§8.3): declarative built-ins first (lowest risk — they were parameter grids anyway), then Panel, then Canvas/IMMERSIVE last. The `switch` shrinks one arm per step; CI stays green after each.
- **`Declarative` built-ins lose their bespoke class entirely** — the host parameter grid from story 301 (§6.2) is their editor. Their DSP stays in `daw-core`; only the `daw-app` view class and `switch` arm are deleted. Wire parameter changes through the Control-Sync single-writer binding / `PluginVM` (**story 294**, on story 291's single-writer discipline; the host editor frame is story 301), not a re-pushed `refreshControls()` (`PluginParameterEditorPanel.java:160-176`) — this is what finally retires the §1.4 feedback bug for the built-ins too.
- **`Panel` built-ins keep their custom layout** but return it as a `javafx.scene.layout.Region` from `createPanel(EditorContext)` instead of being a `BorderPane` stitched in by the controller. The plugin reads/writes through `EditorContext#parameterStore()`, never reaching its own `AudioProcessor` directly (§2.6). `MidSideWrapperPlugin`'s migration aligns with its dedicated view story (story 232) — verify that story's view requirements survive the move to `Panel`.
- **`Canvas`/IMMERSIVE built-ins** (`SpectrumAnalyzerPlugin`, `SoundWaveTelemetryPlugin`, `MatchEqPlugin`, `BinauralMonitorPlugin`) draw into a host-owned `CanvasSurface`; the host drives the loop. These overlap the GPU-canvas display work (`SoundWaveTelemetry` is the story-287 docked `TelemetryView`; `SpectrumAnalyzer` is a story-287 BOTTOM-tile dockable). Coordinate: the Canvas-mode editor and the dockable visualization panel are *different surfaces of the same plugin* — do not duplicate the renderer; the editor's `render(RenderTick)` and the dockable panel can share the underlying display. A floating Canvas editor must gate its animation on `stage.showingProperty()`, not scene presence (`feedback_javafx_hidden_stage_animation_leak.md`).
- **The `host.switchToMasteringView()` arm is dropped, not migrated** — the three mastering plugins (`ParametricEqPlugin`/`CompressorPlugin`/`ReverbPlugin`) have no editor of their own; the `Host` interface keeps `switchToMasteringView()` for the surviving non-`switch` activation path.
- **`SoundWaveTelemetryPlugin`'s arm already routes to the docked panel** (`openSoundWaveTelemetryWindow(...)` → `host.showTelemetryPanel()`, `PluginViewController.java:248-263`, story 287). Its migration to a `Canvas`/IMMERSIVE editor must preserve the story-287 share-one-instance behaviour and the `applyingDockLayout` guard; do not regress the telemetry-float-pop fix.
- **`KeyboardProcessorView` is the bespoke backing view for `VirtualKeyboardPlugin`** (verified, `PluginViewController.java:192`). Its migration to a MINIMAL `Panel` may keep the keyboard widget but must return it as a `Region` and drop the standalone `Stage` path; retire `KeyboardProcessorView`'s window-only assumptions.
- **Source-scan conformance test** for "no `switch` on plugin id" follows the established sentinel methodology (`feedback_ui_overhaul_conformance_sentinel.md`, `feedback_prefer_existing_conventions.md`): reuse the existing source-scan harness shape rather than forking a new one; strip comments, guard against an empty scan, self-exclude the test.
- Files (deleted): the thirteen `daw-app/.../ui/*PluginView.java` classes, `KeyboardProcessorView.java`, `PluginViewController.openBuiltInPluginView()` plus every `open…Window(...)` helper, and `SpectrumDisplayWindow`/`TunerDisplayWindow` where no longer referenced. Files (edited): `daw-app/.../ui/PluginViewController.java` (the `switch` shrinks to nothing; `onActivateBuiltInPlugin` routes through `editorFactory()`), the built-in plugin classes in `daw-core/.../plugin/` (override `editorFactory()`), and `daw-app/.../ui/plugin/EditorFrame.java` (consume the migrated built-ins, no change expected). Files (added): per-built-in `Panel`/`Canvas` implementations where the plugin keeps a custom surface.
- Reference: Plugin View Design Book §1.1/§1.5/§1.9 (problems), §2.3 (SDK is the only seam; `switch` is an internal optimisation being removed), §5.A/§5.C/§5.D, §6.2 (the grid `Declarative` built-ins adopt), §8.3 (this phase, with the exact order and the `open…Window` helper list), §9 rejections #5/#8 (no `switch`, no reflective view discovery); user story 300 (contract), user story 301 (`EditorFrame` + the third-party dispatch path the built-ins join), user story 232 (Mid/Side wrapper view), user story 287 (telemetry/spectrum as dockable panels — share the renderer), user story 079 (built-in discovery via `ServiceLoader`), user story 030 (parameter grid). SKILLs: `javafx-application-design` (§3 Control/Skin, §6 Canvas, §11 threading, §15 anti-patterns), `dawg-annotations-reflection` (ServiceLoader discovery — note the SKILL is stale on sealed-permits), `research-daw`.
