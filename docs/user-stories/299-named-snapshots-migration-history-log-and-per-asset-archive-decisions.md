# Named Snapshots, the Migration History Log, and Per-Asset Archive Decisions

## Motivation

This is **Stage 5** of the Project Manager Design Book §7 migration path — pure additions over the now-robust core (§7 Stage 5: "low risk"). It closes the last three §1 gaps:

- **§1.6 (migration is a one-shot dialog).** `MigrationReportDialog` appears once after a load and offers an in-app "Roll back…" (`ProjectLifecycleController.rollbackMigration(...)`, wired by story 247). But there is no *catalogue* of migrations across the project's life, no *diff* against the pre-migration backup, and no standing sign that the project is on a newer schema than its backup. "A long-lived project will be migrated dozens of times — the user needs a build log, not a one-off pop-up."
- **§1.7 / §3.6 (Snapshot is overloaded and unbuilt as a user feature).** "Snapshot" is used loosely in places to mean a named save, but there is no user-named, never-auto-deleted save distinct from a timed checkpoint. §3.6 fixes this: a Snapshot is a copy of `project.daw` into `snapshots/` — "before mixdown", "client review v3" — that **never auto-deletes**.
- **§4.7 (archive missing-assets is all-or-nothing).** The existing `.dawz` archive worker (`ProjectArchiver.saveAsArchive(...)`, driven by `onArchiveProject` at `ProjectLifecycleController.java:345` — the book's §4.7 cite of `:318` has drifted) handles missing assets as a single yes/no on a background virtual thread. §4.7 makes it **per-asset**: skip / locate manually / use stub, each asset decided independently.

The model and engine are already good; this is UI re-expression plus a small snapshot store. It builds on stories 295-298 (the strip, the Hub, sessions, and the journal are all in place).

## Goals

- **Add named snapshots** (§3.6, §5.1): a `SnapshotStore` in `daw-core` (JavaFX-free) that copies the current `project.daw` to `snapshots/<date>-<slug>.daw` atomically (`.tmp` + fsync + `ATOMIC_MOVE`, §5.2) and lists existing snapshots with name + timestamp + size. Snapshots **never rotate / never auto-delete** (§6.4 — explicit). Creation runs off the FX thread on a virtual thread.
- **Snapshot UI in the Session Manager dock and Save menu** (§4.3 footer "[+ Snapshot…]", §7 Stage 5): a "Snapshot…" action prompts for a name and creates one; snapshot ● rows appear in the dock (story 297) with "Restore from here" and "Compare with current." This is also the surface that finally honours story 190 (Snapshot History Browser) / story 242 (instantiate it) by giving snapshots a real producer and a navigable home.
- **Build the `MigrationHistoryView`** (§4.6.2): a scrollable timeline of every migration the project has undergone, each row showing from→to schema, when, trigger ("Auto on open"), the backup filename, a bullet list of what changed, and **[ Diff ▸ ]** + **[ Roll back ▸ ]** actions. Reachable from the Project Hub (story 296) and the strip's Schema cell (story 295). It **reuses** `MigrationReportDialog` as the per-event row presenter (Appendix A: "Reused as the per-event row; new `MigrationHistoryView` host") and its existing "don't surface again" flag.
- **Diff + roll back per migration** (§4.6.2): "Roll back" calls the existing `ProjectLifecycleController.rollbackMigration(...)` (copy/reload when a backup exists; abandon the in-memory migrated project when it does not — story 247 behaviour, unchanged) for the selected historical backup. "Diff" compares the current `project.daw` against the chosen `project.daw.v<n>.<ts>.bak`. A pre-migration backup is **never deleted while the project still carries its source schema** (§6.4 override) so the roll-back path stays valid indefinitely.
- **Per-asset missing-assets decisions in the Archive dialog** (§4.7): replace the all-or-nothing missing-assets confirmation with the §4.7 dialog — per asset, choose Skip / Locate… / Use silence (or Use stub for presets) — plus include toggles (sessions+takes, checkpoints, journal segments, snapshots), an estimated archive size, and an ETA. Progress flows to the strip footer, **not a modal** (§4.7; §8 "background work without progress"). Reuse `ProjectArchiver`'s `ArchiveHeader` / `MissingAssetResolver` (`daw-core/.../persistence/archive/ProjectArchiver.java`).
- **Archive / restore run under structured concurrency** (§2.6, §5.3): pack and restore fan out (checksum + copy + serialise + write-header; unzip + rewrite-xml + seed-checkpoint) inside `DawScope.openShutdownOnFailure(...)` so any subtask failure cancels the rest and the user sees one error — never a partial archive on disk. Carry context via the explicit `ProjectContext` record (the `ScopedValue` non-preview fallback from story 298 / §5.4).
- **Confirmations only for irreversible actions** (§2.5): overwrite-on-restore, dropping an archive's missing assets, and deleting a checkpoint *history* prompt; creating a snapshot and rolling back to a backup (which itself leaves the backup intact) do not need a modal — they surface Undo where reversible.
- Tests:
  - `SnapshotNeverAutoDeletesTest` (new, `daw-core`): create N snapshots past the checkpoint retention count, run retention, assert all snapshots survive (§6.4); assert atomic-replace on write.
  - `MigrationHistoryViewListsAllBackupsTest` (new): with several `project.daw.v<n>.<ts>.bak` files present, assert one row per migration in order, each linking its backup, reusing `MigrationReportDialog` as the row presenter.
  - `RollBackUsesExistingControllerTest` (new): assert "Roll back" delegates to `rollbackMigration(...)` for the selected backup and that the backup is retained afterward (§6.4 override).
  - `PerAssetDecisionArchiveTest` (new): give three missing assets distinct decisions (Skip / Locate / Use silence), assert the produced `.dawz` honours each independently (not all-or-nothing).
  - `ArchiveRestoreStructuredRollbackTest` (new): fail one restore subtask, assert `DawScope.openShutdownOnFailure` cancels siblings and leaves no partial project on disk.
  - `ArchiveProgressToStripNotModalTest` (new): assert archive progress publishes to `ProjectOperationProgress` (story 295) and that no modal blocks the FX thread during packing (§4.7 / §8).
  - `SnapshotRowCompareBubbleTest` (new): assert a snapshot row's "Compare with current" fires a typed request verified on the payload via a parent `addEventFilter`, not `Event.getSource()` (`feedback_javafx_bubbling_event_test_pitfall.md`).

## Non-Goals

- **No migration *registry* change.** The set of schema migrations and their order (story 188 registry) is untouched; this story only *catalogues and diffs* what already ran. No new migration steps.
- **No new archive *format*.** `.dawz`, `ArchiveHeader`, and `MissingAssetResolver` are reused as-is (story 189 / 245); only the missing-assets *decision UI* and the progress surface change.
- **No checkpoint-history *deletion* feature.** §2.5 lists "delete a checkpoint history" as an irreversible action needing confirmation, but building that destructive action is out of scope; this story only states the confirmation rule for it.
- **No `rollbackMigration` behaviour change.** The copy/reload-vs-abandon logic from story 247 is reused verbatim; this story adds a *history host* around it, not new roll-back semantics.
- **No journal or session model changes.** Stories 297/298 own those; this story consumes sessions (to scope "export today's takes") and the journal (already rotated by checkpoints) without altering them.
- **No automatic copy-back or fallback-workspace policy.** §6.2's ENOSPC fallback is separate disk-contract work; archive here assumes a user-chosen destination.

## Technical Notes

- Files: new `daw-core/.../persistence/snapshot/SnapshotStore.java` (atomic copy + list, never-delete); new `daw-app/.../ui/MigrationHistoryView.java` (host that reuses `MigrationReportDialog` rows), `daw-app/.../ui/ArchiveDialog.java` (or enrich the existing `ArchiveSummaryDialog`) for per-asset decisions; edits to `daw-app/.../ui/ProjectLifecycleController.java` (snapshot action; `onArchiveProject` at :345 → per-asset flow + strip progress; `rollbackMigration(Path, MigrationReport)` at :701 reused for roll-back; `MigrationHistoryView` entry points). Snapshots live in `snapshots/` per the §3.2 layout.
- Reuse, don't reinvent: `ProjectArchiver` + `ArchiveHeader` + `MissingAssetResolver` (`daw-core/.../persistence/archive/ProjectArchiver.java`) and `MigrationReportDialog` (`daw-app/.../ui/MigrationReportDialog.java`) are the existing-good pieces (§1.10); the story is a UI re-expression of capabilities already in the model (Appendix A closing note). `rollbackMigration(...)` stays in `ProjectLifecycleController` (story 247).
- Structured concurrency: archive/restore use `DawScope.openShutdownOnFailure("archive-pack")` / `("archive-restore")` with `fork(...)` per subtask + `joinAll()` (verified API in `daw-core/.../concurrent/DawScope.java`); on any failure the scope cancels and rolls back — no partial `.dawz` (§2.6 / §5.3). Context flows via the explicit `ProjectContext` record (the §5.4 non-preview fallback; `ScopedValue`/JEP 506 is the target once it exits preview — do not require `--enable-preview` for the shipping path).
- Progress, never modal: archive packing publishes into `ProjectOperationProgress` (story 295) so the strip footer shows it; the §4.7 dialog gathers decisions then dismisses, and the long pack runs in the background (§8 "background work without progress" / "no modal for reversible work"). `DawgDialog` chrome (story 276) for the dialogs.
- Snapshots are full copies of `project.daw` (§3.6) written atomically off the FX thread (virtual thread; `javafx-application-design` §11). Retention: snapshots are excluded from `BackupRetentionPolicy` rotation (§6.4); pre-migration backups get the §6.4 override (never deleted while source schema is still carried).
- `daw-core` stays JavaFX-free: `SnapshotStore` is plain Java; views/dialogs live in `daw-app` (`project_daw_app_non_modular.md`). Tokens not hex for any new chrome; tint glyphs from resolved CSS (`feedback_iconnode_tint_from_resolved_css.md`). Place toolkit tests in `...app.ui` (`project_extendwith_jpms_test_env.md`) and extract pure transforms for off-toolkit tests (`feedback_javafx_headless_test_pitfalls.md`).
