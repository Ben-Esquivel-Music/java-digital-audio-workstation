# Write-Ahead Journal and the Crash Recovery Dialog

## Motivation

This is **Stage 4** of the Project Manager Design Book ¬ß7 migration path ‚Äî the stage that turns "crashes are routine, not exceptional" (¬ß2.3) from a slogan into a guarantee. It fixes the deepest ¬ß1 problems:

- **¬ß1.5 (the dirty-bit is a single boolean).** `host.isProjectDirty()` is one `boolean`. It cannot answer "what did I change since the last save?", "was that last change recorded?", or "did the autosave 90 seconds ago succeed?" The codebase has the plumbing (`UndoManager`) but never connects it to persistence. The journal connects them.
- **¬ß1.2 / ¬ß2.3 (the 30 s gap between checkpoints).** Even with `AutoSaveConfig.LONG_SESSION` at 30 s, work between checkpoints is unprotected. The ¬ß5.1 write-ahead journal absorbs **every state mutation instantly** on a background thread, so a crash loses seconds, not minutes.
- **Today crashes silently re-load the last `project.daw`** (Appendix A: "Recovery dialog ‚Äî none; crashes today silently re-load last `project.daw`"). Work since the last save is simply gone, with no notice.

The ¬ß5.1 three-layer model is: **journal** (append-only, fsync per event group, 16 MB rotation) ‚Üí **checkpoint** (timed, subsumes the prior segment) / **snapshot** ‚Üí **`project.daw`** (atomic head). The ¬ß4.6.1 Recovery dialog, on a launch that follows an unclean exit, offers "Recover everything (replay journal onto last checkpoint)" / "Restore checkpoint only" / "Inspect events," replaying inside a `DawScope.openShutdownOnFailure` so recovery is never half-applied. Per ¬ß7 Stage 4 this ships **behind a `Use journaled persistence` preference for one release** because it changes the crash-recovery story substantially.

## Goals

- **Add a write-ahead journal in `daw-core`** (¬ß3.7, ¬ß5.1): an append-only `journal/segment-NNN.bin` log. Every undoable state mutation (`UndoManager` apply/undo) enqueues a **compact record** to a bounded queue; a **single dedicated journal-writer virtual thread** drains it, appends, and `fsync()`s a group. Segments rotate on every successful checkpoint (the new checkpoint subsumes the prior segment) or at **16 MB** (¬ß3.7, ¬ß6.4).
- **The FX thread never blocks on the journal** (¬ß5.3, ¬ß6.3, book ¬ß2.1). The FX thread enqueues with a **non-blocking `offer()`**; if the queue is full (slow disk) the event spills into an in-memory **overflow buffer bounded at 64 MB** that the writer drains when I/O resumes. If the overflow fills, escalate to a red "Save your work ‚Äî disk unresponsive" strip alert and force the next user-initiated save to a synchronous flush on a **background** thread (never FX). **Events are never silently dropped** (¬ß6.3 ‚Äî explicitly forbidden).
- **Hook the journal to `UndoManager`** (¬ß7 Stage 4): apply/undo actions emit journal records. Reuse the story 283 `EventBus` producers as the change source where they already fire (`project_eventbus_dormant.md`); the journal record is a compact, replayable descriptor, not the full project XML (that is the checkpoint's job ‚Äî ¬ß1.1's "full re-serialise on every Save" is exactly what the journal avoids).
- **Build the `RecoveryDialog`** (¬ß4.6.1) shown when a recoverable project is opened from the Welcome Recover region (story 296): show last clean save (checkpoint #N + time) and last journal event (type + time), the count of replayable minutes, and three options ‚Äî replay-onto-checkpoint / checkpoint-only / inspect-events ‚Äî plus a "Show technical detail" expander (segment file, size, event types). The dialog is a `DawgDialog` (story 276 chrome).
- **Replay under structured concurrency** (¬ß4.6.1, ¬ß5.3, ¬ß2.6): the replay runs on virtual threads inside `DawScope.openShutdownOnFailure(...)` (`daw-core/.../concurrent/DawScope.java`) so checksum + load-checkpoint + replay-segment either all succeed or roll back ‚Äî partial recovery is never half-applied. Carry the active project context (path, request id) without `ThreadLocal`: prefer `ScopedValue` (JEP 506, second preview in JDK 26 ‚Äî requires `--enable-preview`), with the **explicit `ProjectContext` record threaded through `DawScope` fork lambdas as the non-preview shipping fallback**, exactly as ¬ß2.6 / ¬ß5.4 specify.
- **Drive the strip's Journal cell** (¬ß4.4, story 295): `journalEventsQueued` on `ProjectOperationProgress` reflects events since the last checkpoint; the ¬ß6.3 back-pressure warnings ("Disk is slow ‚Äî events buffering in memory", then the red escalation) surface through the same model. This replaces the ¬ß8 "single boolean dirty flag driving the title-bar dot" with "events queued since last checkpoint."
- **Ship behind a preference** (¬ß7 Stage 4): a `Use journaled persistence` setting (default off for the first release) gates journal writes and the recovery path; when off, behaviour is exactly today's (checkpoint-only). Document the one-release rollout.
- Tests (mostly `daw-core`, JavaFX-free):
  - `JournalAppendFsyncRotationTest`: append records, assert a single writer thread drains the queue, fsyncs groups, and rotates the segment at 16 MB and on checkpoint success.
  - `FxOfferNeverBlocksTest`: fill the bounded queue, assert `offer()` returns immediately (FX-thread proxy never blocks) and events spill to the 64 MB overflow rather than blocking or dropping (`feedback_ffm_tests_no_non_windows_assumptions.md` ‚Äî no non-Windows assumptions; user's primary platform is Windows).
  - `OverflowEscalationNoDropTest`: overflow the 64 MB buffer, assert the red escalation fires and that **no event is dropped** (count in == count eventually persisted); assert the forced flush runs on a non-FX thread.
  - `RecoveryReplayStructuredTest`: corrupt one replay subtask, assert the `DawScope.openShutdownOnFailure` cancels siblings and leaves no half-applied state (atomic rollback).
  - `ProjectContextNoThreadLocalTest`: assert request context propagates to forked workers without any `ThreadLocal` (source-scan + behavioural), satisfying the ¬ß2.6 / ¬ß8 "no `ThreadLocal`" constraint regardless of preview status.
  - `RecoveryDialogOptionsTest` (`daw-app`): with a replayable segment present, assert the three options render with correct last-save / last-event figures; place the toolkit test in `...app.ui` (`project_extendwith_jpms_test_env.md`).
  - `JournaledPersistencePreferenceGateTest`: with the preference off, assert no journal segment is written and recovery is checkpoint-only; with it on, assert segments appear.

## Non-Goals

- **No full CoW versioned-tree checkpoint refactor.** ¬ß5.3's copy-on-write snapshot-scaling strategy (and its dirty-flag fallback) is a checkpoint-writer optimization tracked separately; this story keeps the existing `CheckpointManager` serialise-on-virtual-thread path and only adds the journal in front of it. (`scope fix, defer tangential finds`.)
- **No snapshots, migration log, or per-asset archive.** Those are Stage 5 / story 299. The journal *enables* richer recovery but this story stops at replay-onto-checkpoint.
- **No `sessions/` manifest creation.** Story 297 owns sessions; this story populates a session's journal-segment links *if* manifests exist, and otherwise journals against the bare project.
- **No fallback-workspace ENOSPC policy.** The ¬ß6.2 low-disk fallback workspace (and its blocking modal / copy-back) is disk-space-contract work, separate from the journal's own back-pressure; the journal's job here is to never block and never drop, not to relocate checkpoints.
- **No removal of the boolean dirty bit's callers.** The strip's Journal cell supersedes it for the user, but ripping out every `isProjectDirty()` caller is out of scope (`test-only callers aren't live usage` applies if any are found stranded ‚Äî flag, don't bundle).

## Technical Notes

- Files: new `daw-core/.../persistence/journal/JournalWriter.java` (single writer vthread + bounded queue + 64 MB overflow), `JournalSegment.java`, `JournalRecord.java`, `JournalReplayer.java` (structured replay), `ProjectContext.java` (the ¬ß5.4 non-preview context record); new `daw-app/.../ui/RecoveryDialog.java` (`DawgDialog`); edits to `UndoManager` hookup and `daw-app/.../ui/ProjectLifecycleController.java` (recovery entry from the Welcome Recover region, story 296). Journal lives in `journal/` per the ¬ß3.2 layout.
- Structured concurrency: replay uses `DawScope.openShutdownOnFailure("project-recover")` then `fork(...)` per subtask + `joinAll()` (verified API: `openShutdownOnFailure(String)`, `fork(Callable)`, `joinAll()`, `Subtask.resultNow()`, `close()` in `daw-core/.../concurrent/DawScope.java`). The same wrapper backs archive/restore (story 299).
- Preview-feature discipline: `ScopedValue.runWhere(...)` is the *target* for request context (`java-26.agent.md` Project Loom; JEP 506 second preview, `--enable-preview`); the shipping path is the explicit `ProjectContext` threaded through fork lambdas ‚Äî a single-commit mechanical swap once the JEP finalizes (¬ß2.6 / ¬ß5.4 note). Either way: **no `ThreadLocal`** (¬ß8).
- Atomicity: every journal segment, checkpoint, and `project.daw` write is `.tmp` + fsync + `Files.move(ATOMIC_MOVE)` (¬ß5.2). A crash mid-write leaves the prior file intact.
- Threading correctness: the journal-writer thread is the *only* writer per segment (one writer per file, ¬ß5.3); any shared mutable state it touches uses `ReentrantLock`, never `synchronized` around blocking I/O on a virtual thread (carrier-thread pinning silently throttles throughput ‚Äî `java-26.agent.md` Project Loom; ¬ß8 rejection). Publish-side-effects-before-terminal-state ordering applies to the writer's ack path (`feedback_publish_side_effects_before_terminal_state.md`).
- `daw-core` stays JavaFX-free: journal, replayer, and context record are plain Java; only `RecoveryDialog` and the strip wiring touch JavaFX (`project_daw_app_non_modular.md`). Windows is the primary target ‚Äî FFM/file tests must not assume non-Windows behaviour (`feedback_ffm_tests_no_non_windows_assumptions.md`).
