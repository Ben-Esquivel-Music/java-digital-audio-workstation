# Absorb the Audio Device Dialog into the Audio Category

## Motivation

The Settings View Design Book exists, more than anything, to delete one pattern: a dialog that opens another dialog. Today the Audio **tab** inside `SettingsDialog` carries an "Audio Device Settings…" button (`openAudioDeviceDialogButton`, verified at `SettingsDialog.java:306-310`, whose `setOnAction` calls `openAudioDeviceDialog()` at line 320) that launches a *second* modal, `AudioSettingsDialog` — a dialog, inside a tab, inside a dialog (book §1.1 / rejection #1). Stage 3 (§7) folds that second dialog's controls into the Audio category's Device and Performance groups of the Rail & Pane shell built in story 306, and removes the button.

This resolves the book's headline problems:

- **§1.1 / rejection #1 — the nested-modal anti-pattern.** "Audio Device Settings…" stops being a button that opens a window and becomes groups you navigate to (§2.1 one front door). This is the single pattern the whole book was written to remove.
- **§1.2 — the model is wider than the view.** `AudioSettingsDialog` is the *only* place `mixPrecision`, `audioBackend`, `audioInputDevice`, `audioOutputDevice`, `applyLatencyCompensation`, `srcQuality`, `workerPoolSize`, `masterCpuBudgetFraction`, `masterCpuBudgetPolicy` (book §1.2) are reachable. Story 305 catalogued them; this story finally gives each a *row* in the one surface, so there is no longer a split catalogue (rejection #5: a setting is either user-facing with a row, or internal).
- **§1.6 / §6.4 — the honest apply contract, and the death of the blanket restart hint.** Today the audio tab carries an always-on "Changes to audio settings may require a restart." hint (verified `SettingsDialog.java:312`, styled with the inline `#ff9100` literal at line 314 — book rejection #4) that cries wolf on every visit. With the story-305 apply-class metadata, most audio settings are `ENGINE_RECONFIGURE` (sample rate, buffer size, SRC quality, worker pool) and re-arm the engine off-thread on Apply; only `audioBackend` is `RESTART_REQUIRED`. The §5.6 banner now appears *only* when the backend is actually dirty, and names it.
- **§2.7 / §6.6 — heavy work off the FX thread.** `AudioSettingsDialog` enumerates devices today (its imports include `Platform`, `Timeline`, and an `AudioDeviceEvent` listener); enumeration is I/O and must run on a background virtual thread with a progress affordance, marshalling only the result back with `Platform.runLater` (book §2.7/§6.6).

Per §7 Stage 3, this is the first stage that *removes* a legacy surface; the Recording and Backups dialogs are absorbed separately in story 308.

## Goals

- **Audio category gains Device and Performance groups (§3.3).** The Audio category in the story-306 rail renders three groups — Format (sample rate, bit depth, buffer size), Device (backend, input device, output device, latency compensation), and Performance (SRC quality, worker pool size, master CPU budget) — each as generic setting rows (§5.4) driven by the story-305 descriptors. Every control currently in `AudioSettingsDialog` has a row; no audio setting remains reachable only through a separate window.
- **Remove the "Audio Device Settings…" button and the nested modal (§2.1, rejection #1).** Delete `openAudioDeviceDialogButton` (`SettingsDialog.java:306-310`) and the `openAudioDeviceDialog()` path (line 320) that constructs and shows `AudioSettingsDialog` from inside the settings surface. The Audio category *is* the device settings now. Per `feedback_test_only_callers_are_not_live_usage.md`, if `AudioSettingsDialog`'s only remaining callers are its own tests (`AudioSettingsDialogTest` is verified to exist), remove the class and migrate any still-valid assertions onto the catalogue/row path; if it is still reachable from another live entry point (e.g. a toolbar/menu action — verify in `MainController`), keep the class but route it to deep-link into the Audio category rather than open a modal.
- **Honest apply classes, with the restart banner (§6.4, §5.6).** Sample rate, buffer size, SRC quality, and worker pool size carry the `engine-reconfigure` badge and, on Apply, re-arm the audio engine off-thread (the existing engine-reconfigure path, story 098 / `AudioEngineController`) — not a restart. `audioBackend` carries the `restart-required` badge and, when dirty, raises the §5.6 banner naming "Audio backend" (replacing the always-on `SettingsDialog.java:312` hint, rejection #4). The footer's contextual `warn` note (§5.5) appears only when an engine-reconfigure or restart setting is pending.
- **Device enumeration on a background virtual thread (§2.7, §6.6).** Backend / input-device / output-device choice lists are populated by enumerating the audio backend on a background virtual thread, with a small progress affordance in the Device group header (the "◷ recomputing…" idiom from §5.8) while a slow backend enumerates; the result is applied to the choice controls via `Platform.runLater`. The settings surface never blocks on enumeration. Use structured concurrency so a cancelled/closed surface tears the work down cleanly (`java-26.agent.md` Project Loom).
- **Per-group reset arrives (§5.7 tier 2).** Each Audio group header gets a "Reset group" affordance that resets every setting in that group to its descriptor default (building on story 306's per-setting `↺`). This is §5.7 tier 2; per-category and restore-all follow in stories 308/309.
- **Tokens, never literals (§2.6).** The audio rows, the engine-reconfigure/restart badges, and the banner use semantic tokens (`warn` for the restart badge/banner, `text-muted` for descriptions); the inline `-fx-text-fill: #ff9100; -fx-font-size: 10px;` restart hint (`SettingsDialog.java:314`, rejection #2) is gone with the button it annotated.
- Tests:
  - `AudioDeviceDialogButtonRemovedTest` (new): the settings surface exposes no "Audio Device Settings…" button and no code path that shows `AudioSettingsDialog` as a nested modal (the rejection #1 guard).
  - `AllAudioSettingsHaveRowsTest` (new): every `SettingsModel` audio key named in §1.2 (`mixPrecision`, `audioBackend`, `audioInputDevice`, `audioOutputDevice`, `applyLatencyCompensation`, `srcQuality`, `workerPoolSize`, `masterCpuBudgetFraction`, `masterCpuBudgetPolicy`) renders a setting row in the Audio category (closes the §1.2 split-catalogue gap visibly).
  - `AudioApplyClassesTest` (new): sample rate / buffer / SRC / worker pool show the `engine-reconfigure` badge and do **not** trigger a restart banner; `audioBackend` shows the `restart-required` badge and, when dirty, raises the §5.6 banner naming it; no always-on restart hint is present when nothing restart-class is dirty (the rejection #4 guard).
  - `DeviceEnumerationOffFxThreadTest` (new): device enumeration runs on a non-FX thread and applies results via `Platform.runLater`; assert the enumeration callable is not invoked on the FX thread and that a slow enumeration does not block construction (substitute a fake slow backend). Capture+rethrow any assertion made inside an FX runnable (`feedback_javafx_headless_test_pitfalls.md`).
  - `AudioGroupResetTest` (new): "Reset group" on the Performance group restores SRC/worker-pool/CPU-budget to defaults without touching Device or Format settings.
  - `EngineReconfiguresOnApplyTest` (new): applying a changed buffer size invokes the engine-reconfigure path (story 098) off the FX thread, not a restart; applying a changed backend does not reconfigure live but arms the restart banner.

## Non-Goals

- **No new `SettingsModel` keys and no engine API change.** Every audio setting already exists in `SettingsModel` (that is the §1.2 point); this story gives them rows and an honest apply class. The engine-reconfigure path (story 098) is reused, not extended.
- **No absorbing of Recording or Backups.** `MetronomeSettingsDialog` and `BackupSettingsDialog` keep their entry points until story 308. This stage is audio only (§7 Stage 3).
- **No scope chip yet.** Audio settings are `AUDIO_DEVICE` scope (story 305 metadata, §3.2) but the on-screen scope tag in the pane header (§5.3) is surfaced uniformly in story 309. The apply-class badge ships here; the scope chip is 309.
- **No import/export, no restore-all, no first-run wizard.** Story 309 (§5.9, §5.7 tier 4, Concept D). Per-group reset (tier 2) is the ceiling for this story.
- **No ASIO-specific sample-rate negotiation work.** That is its own concern (story 258); this story renders the existing sample-rate control as a row and routes its Apply through the existing engine-reconfigure path — it does not change how a backend negotiates a rate.
- **No embedded-panel promotion.** The surface stays modal (§4.B deferred).

Mirror the repo's existing `TrackFreezeController` / `ProjectLifecycleController` off-thread pattern (book §2.7). Per `feedback_ffm_tests_no_non_windows_assumptions.md`, do not write enumeration tests that assume a backend is absent on non-Windows — the user's primary target is Windows + ASIO + multi-channel USB (`user_hardware_environment`); use a fake backend in tests.
