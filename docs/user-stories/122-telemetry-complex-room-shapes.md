# Complex Room Shapes: Domed, Vaulted, and Non-Rectangular Ceilings

## Motivation

The Sound Wave Telemetry engine currently models every room as a rectangular box via the `RoomDimensions(width, length, height)` record. That is a reasonable first approximation for bedrooms and control rooms, but it is a poor model for the spaces where acoustic behavior matters most: churches with domed or barrel-vaulted ceilings, concert halls with shoebox-plus-curved-rear geometries, home theaters with cathedral ceilings, and live rooms with angled "cloud" ceilings designed to diffuse early reflections. In all of those spaces the ceiling geometry is the dominant acoustic feature — a dome focuses reflections to a "hot spot," a barrel vault produces strong comb filtering along its axis, and an angled ceiling is specifically engineered to avoid the flutter echoes that the current model cannot even represent.

This gap is visible in the existing product in two places. First, `RoomPreset` has entries for room *types* like Concert Hall and Church (see README and story 045) but they all collapse to a single rectangular box, so a church and a shoebox studio of the same volume produce identical telemetry output. Second, the suggestions generated by `SoundWaveTelemetryEngine` cannot reason about ceiling shape at all — it can tell a user they are too close to the left wall, but not that they are standing at the focal point of a dome. Adding first-class support for non-rectangular ceilings lets the visualization, the RT60 math, and the suggestions all become meaningful in the spaces where acoustic analysis is most valuable.

The engineering cost is bounded. `RoomDimensions` is already a record, so extending the model without breaking existing callers is straightforward, and `RoomTelemetryDisplay` is a 2D top-down view that can render ceiling shape as an overlay (shaded contour or iso-height lines) rather than requiring a full 3D view. The heavy lifting is in the engine: `SoundWaveTelemetryEngine` already computes wall reflections, and extending it to compute curved-surface reflections via faceted approximation is a well-known technique in room acoustics.

## Goals

- Introduce a `CeilingShape` sealed interface in `daw-sdk` (`com.benesquivelmusic.daw.sdk.telemetry`) with the following permitted variants, each as a record so they compose naturally with the existing `RoomDimensions`:
  - `CeilingShape.Flat(double height)` — the current behavior, kept as the default for backward compatibility
  - `CeilingShape.Domed(double baseHeight, double apexHeight)` — a spherical-section dome rising from `baseHeight` at the walls to `apexHeight` at the room center
  - `CeilingShape.BarrelVault(double baseHeight, double apexHeight, Axis axis)` — a cylindrical vault running along the room's width or length
  - `CeilingShape.Cathedral(double eaveHeight, double ridgeHeight, Axis ridgeAxis)` — a symmetric pitched ceiling with a central ridge
  - `CeilingShape.Angled(double lowHeight, double highHeight, Axis slopeAxis)` — a single-slope "cloud" ceiling
- Replace the single `height` parameter on `RoomDimensions` with a new constructor `RoomDimensions(double width, double length, CeilingShape ceiling)`, keeping the existing `RoomDimensions(double width, double length, double height)` constructor as a convenience that wraps `CeilingShape.Flat`; add a `ceiling()` accessor and have `height()` return the *maximum* height of the shape so existing read-only callers keep working
- Update `RoomDimensions.volume()` and `surfaceArea()` to integrate over the actual ceiling shape (closed-form for each permitted variant — no numerical integration needed), so RT60 reflects the real volume of a dome or vault rather than a bounding box
- Extend `SoundWaveTelemetryEngine` to compute ceiling reflections against the actual `CeilingShape`:
  - Curved surfaces (dome, barrel) are approximated by a configurable number of facets; each facet reflects as a flat panel
  - Detect and emit a `TelemetrySuggestion` when a microphone sits near the geometric focus of a `Domed` or `BarrelVault` ceiling (classic "whispering-gallery" hot spot)
  - Emit a suggestion when a source/mic pair is aligned with a `Cathedral` ridge axis (flutter-echo risk)
- Extend `TelemetrySetupPanel` with a "Ceiling shape" section:
  - A `ComboBox<CeilingShape.Kind>` to pick Flat / Domed / Barrel Vault / Cathedral / Angled
  - Contextual numeric inputs that appear for the selected variant (e.g., apex height for Domed; ridge axis for Cathedral)
  - Live RT60 preview updates as ceiling parameters change
  - An iso-contour or shaded overlay on `RoomTelemetryDisplay` so the user can see the ceiling profile on the 2D canvas (lighter = higher, darker = lower), with a small side-view silhouette inset for the non-axisymmetric shapes
- Extend `RoomPreset` / `RoomPresetLibrary` so presets that logically have a non-flat ceiling (Concert Hall, Church, Cathedral) declare the appropriate `CeilingShape` instead of defaulting to flat
- Update project persistence (`ProjectSerializer` / `ProjectDeserializer`) to round-trip the new `CeilingShape` data; files written by previous versions (flat height only) continue to load by materializing `CeilingShape.Flat`
- Unit tests cover: each `CeilingShape` variant validates its inputs (no negative heights, apex ≥ base, etc.); `volume()` and `surfaceArea()` match known closed-form values for each shape; legacy `RoomDimensions(w,l,h)` still produces a `Flat` ceiling; the engine emits a focus-point suggestion when a mic is placed under a dome's apex; persistence round-trips every variant; a project file written before this change still loads and opens with a flat ceiling

## Non-Goals

- Arbitrary user-drawn ceiling meshes or imported CAD geometry — this story is restricted to the five parameterized variants above, which cover the overwhelming majority of real rooms
- Non-rectangular floor plans (L-shaped rooms, fan-shaped halls) — the floor remains a rectangle defined by `width` and `length`; floor-plan shape is a separate future story
- A full 3D rendered view of the room — the telemetry canvas remains 2D top-down with an iso-contour ceiling overlay and a small side-silhouette inset; 3D visualization is out of scope
- Material variation across the ceiling (e.g., absorbent panels only on the dome) — the single `WallMaterial` on `RoomConfiguration` continues to apply to all surfaces in this story; per-surface materials are a separate future story
- Changing the `FdnRoomSimulator` (story 045) algorithm itself — only the geometry inputs it consumes are extended here
